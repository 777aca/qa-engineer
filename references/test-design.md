## 1. 从需求到用例：一般流程

1. **读懂需求**：通读 PRD/UI 稿/接口文档，列出"这个功能究竟要做什么"
2. **提炼测试点**：每一条业务规则、每一个输入字段都是一个测试点候选
3. **选择设计方法**：参考下表
4. **交叉覆盖**：每个测试点至少被一条用例覆盖；高风险点用多方法覆盖
5. **评审**：与开发/产品一起评审用例，补充遗漏

| 场景特征 | 推荐方法 |
|---------|---------|
| 输入有明显分组 / 有效与无效 | 等价类划分 |
| 输入是数值或长度受限 | 边界值分析 |
| 多个条件组合决定输出 | 判定表 |
| 有业务流程、主路径/分支路径 | 场景法 |
| 对象有状态机（如订单、账号） | 状态迁移 |
| 多参数两两组合 | 正交实验 |
| 成熟功能找缺陷 | 错误推测 |

## 2. 等价类划分

把输入空间划分成若干个**等价类**：在同一类中任选一个样本，测试效果等同。

**步骤**：
1. 识别输入条件
2. 划分**有效等价类**（EC）和**无效等价类**（IC）
3. 每个有效类选一个代表值、每个无效类独立出一条用例

**示例（年龄字段，0~150）**：

| 分类 | 类型 | 代表值 |
|-----|-----|-------|
| 0~150 | EC | 30 |
| < 0 | IC | -1 |
| > 150 | IC | 151 |
| 非数字 | IC | "abc" |
| 空值 | IC | "" |

## 3. 边界值分析

绝大多数 Bug 发生在边界。对每个有范围的输入，至少测：

- **最小值 - 1**（非法）
- **最小值**（合法）
- **最小值 + 1**（合法）
- **最大值 - 1**（合法）
- **最大值**（合法）
- **最大值 + 1**（非法）

**示例（密码 8~16 位）**：
`7 位`（非法）、`8 位`（合法）、`9 位`（合法）、`15 位`（合法）、`16 位`（合法）、`17 位`（非法）

**常见易忽略边界**：
- 零、负数、空字符串、null、undefined
- 数组长度 0、1、最大
- 分页第 0 页、第 1 页、最后一页、最后一页+1
- 时间边界：月初/月末、跨年、闰年 2-29、时区切换
- 整数溢出：Int32 边界、Number.MAX_SAFE_INTEGER

## 4. 判定表法

当输出由多个条件组合决定时使用。

**步骤**：
1. 列出所有条件（Condition）
2. 列出所有动作（Action）
3. 写出规则矩阵（2ⁿ 列，n = 条件数）
4. 合并等价列，得到最终用例

**示例（订单满减）**：

| 条件 | R1 | R2 | R3 | R4 |
|-----|----|----|----|----|
| 金额 ≥ 100 | Y | Y | N | N |
| 会员 | Y | N | Y | N |
| 动作：打 8 折 | ✓ | | | |
| 动作：减 10 元 | | ✓ | | |
| 动作：无优惠 | | | ✓ | ✓ |

## 5. 场景法（用户故事路径）

从用户视角梳理业务流：主流程（基本流）+ 分支流（备选流）。

**示例（电商下单）**：
- 基本流：选商品 → 加购物车 → 结算 → 选地址 → 选支付 → 支付成功
- 备选流：
  - 库存不足
  - 优惠券不可用
  - 支付超时
  - 支付失败
  - 地址缺省

每条路径对应一个用例。

## 6. 状态迁移法

适用于带状态机的业务对象（订单、账号、审批单、会议室预订等）。

**步骤**：
1. 画出状态迁移图：节点 = 状态，边 = 触发事件
2. 为每条迁移路径写一条用例
3. 为每个**非法迁移**写一条用例（如"已完成订单不能取消"）

**示例（订单状态）**：
```
待支付 --[支付]--> 已支付 --[发货]--> 已发货 --[确认收货]--> 已完成
   |                   |                |
   [取消]              [退款]           [申请售后]
   v                   v                v
 已取消              已退款           售后中
```
用例覆盖：每条实线箭头 + 非法路径（如"已完成 → 取消"应拒绝）。

## 7. 正交实验法

多参数组合爆炸时使用，用正交表大幅降低组合数。

**示例**：4 参数各 3 取值 → 全组合 81 条；用 L9(3⁴) 正交表只需 9 条即可覆盖所有两两组合。

工具推荐：[allpairs](https://www.satisfice.com/download/allpairs)、`pytest-pairwise`、在线生成器。

## 8. 错误推测法

基于经验直接推测可能出错的点。常见清单：
- 输入：空、极长、特殊字符（`'"<>&\n\0`）、SQL 注入、XSS、unicode、emoji、RTL
- 并发：同时点两次提交按钮、双开浏览器同账号操作
- 网络：断网、慢网、返回 500、返回超时、返回格式错误
- 时序：前端还在 loading 时用户点了返回
- 权限：未登录访问、越权访问（用 A 的 token 访问 B 的数据）
- 兼容：不同浏览器、不同 OS、旧版本客户端

## 9. 用例优先级与风险

每条用例都应标注优先级（P0~P3）：

| 优先级 | 含义 | 何时执行 |
|-------|------|---------|
| P0 | 核心主流程，挂了整个系统不可用 | 每次冒烟必跑 |
| P1 | 主要功能，挂了影响重要业务 | 每次回归必跑 |
| P2 | 次要功能/分支场景 | 重要版本回归 |
| P3 | 极端场景、兼容性 | 按需执行 |

风险导向：**高概率 × 高影响 = 高优先级**。

## 10. 用例书写模板

```markdown
## TC-{模块}-{编号} {一句话标题}

- **优先级**：P0 / P1 / P2 / P3
- **所属模块**：登录 / 下单 / 支付 / ...
- **前置条件**：
  - 已开通测试账号 `test@example.com`
  - 账户余额 > 100 元
- **测试步骤**：
  1. 打开登录页
  2. 输入账号与正确密码
  3. 点击登录
- **预期结果**：
  - 2 秒内跳转至首页
  - 右上角显示用户昵称
  - Cookie 中存在 `sid` 字段
- **实际结果**：（执行时填写）
- **结果**：Pass / Fail
- **备注**：关联需求 PRD-123；回归版本 v1.2
```

```markdown
# {项目名} 测试计划 v{版本}

- 测试层级：单元 / 接口 / UI / 性能
- 测试类型：功能 / 回归 / 兼容 / 安全
- 测试环境：dev / staging / prod-like

## 4. 资源与环境
- 人力
- 设备（手机型号、浏览器版本清单）
- 测试数据来源

## 5. 风险与应对
| 风险 | 概率 | 影响 | 应对 |
|-----|-----|-----|-----|

## 6. 准入准出标准

## 7. 交付物清单
- 测试用例、测试报告、Bug 清单、回归清单
```
