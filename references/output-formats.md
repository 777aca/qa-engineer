## 1. 格式选择决策表

| 产物类型 | 推荐格式 | 为什么 |
|---------|---------|-------|
| **测试用例（给其他测试同事评审）** | Excel (.xlsx) | 团队约定俗成，可批量编辑 |
| **测试用例（给自己理清思路）** | XMind (.xmind) 或大纲 | 层级清晰，易调整 |
| **测试用例（评审会议演示）** | XMind (.xmind) | 逐级展开演示效果最好 |
| **测试用例（导入禅道/TestLink）** | Excel (.xlsx) 按目标格式 | 禅道官方支持 xlsx 导入 |
| **测试用例（git 版本化）** | Markdown | 可 diff 可 review |
| **Bug 报告（提交 JIRA/禅道）** | Markdown（Jira/禅道自带编辑器支持） | 通用 |
| **Bug 报告（邮件/IM 发送）** | Markdown 或 Word | Word 更正式 |
| **测试方案/计划（归档）** | Word (.docx) | 公司通常要求 |
| **测试报告（团队汇报）** | PPT (.pptx) | 可视化，适合会议 |
| **测试报告（给个人/邮件）** | Word (.docx) 或 Markdown | 按收件人习惯 |
| **自动化脚本** | 对应语言源文件 | 无选择 |

## 2. 结构化用例数据模型

**所有用例相关的输出都应以统一的数据结构为源**（然后转换到不同格式）。这保证 xlsx / xmind / markdown 内容一致。

### 标准 JSON / YAML 结构

```yaml
project: 订单模块测试
version: v1.2.0
author: 测试同学
created: 2026-04-21

cases:
  - id: TC-ORDER-001
    module: 订单创建
    sub_module: 正常下单
    priority: P0
    title: 使用默认地址+默认支付方式下单成功
    preconditions:
      - 已登录普通用户账号
      - 购物车内有至少 1 件有库存商品
    steps:
      - 进入购物车页
      - 勾选 1 件商品，点击"去结算"
      - 在订单确认页使用默认地址、默认支付方式
      - 点击"提交订单"
    expected:
      - 跳转到支付页，显示订单号
      - 订单号格式为 O + 14 位数字
      - 订单状态为"待支付"
    case_type: 功能测试    # 功能 / 接口 / 性能 / 兼容 / 安全
    design_method: 场景法  # 等价类 / 边界值 / 判定表 / 场景法 / 状态迁移 / 正交 / 错误推测
    tags: [smoke, regression]
    related_req: PRD-456
```

### 必填字段
`id`, `module`, `priority`, `title`, `steps`, `expected`

### 可选字段
其他都是增强信息，生成不同格式时按需使用。

### Claude 的工作方式
1. 先和用户沟通清楚需求
2. 先产出"中间结构化 JSON/YAML"（或内存中的 Python 列表）
3. 再根据用户选择的格式，调用不同脚本/工具生成最终文件
4. **永远不要"只写 markdown 又复制一份改成 xlsx 格式"**——保持单一数据源

## 3. Markdown 输出

最轻量的格式，**任何任务的默认兜底格式**。

### 单条用例模板
见 `scripts/new_test_case.py` 生成的结构。

### 批量用例列表（推荐结构）
```markdown
# 订单模块测试用例 v1.2.0

> 作者：{author}  创建时间：{date}

## 订单创建 / 正常下单

### TC-ORDER-001【P0】使用默认地址+默认支付方式下单成功

**前置条件**
- 已登录普通用户账号
- 购物车内有至少 1 件有库存商品

**测试步骤**
1. 进入购物车页
2. 勾选 1 件商品，点击"去结算"
3. 在订单确认页使用默认地址、默认支付方式
4. 点击"提交订单"

**预期结果**
- 跳转到支付页，显示订单号
- 订单号格式为 O + 14 位数字
- 订单状态为"待支付"

---
```

## 4. Excel (.xlsx) 输出

### 优先方案：调用本仓库脚本
```bash
python scripts/cases_to_xlsx.py cases.yaml -o output/订单测试用例.xlsx
```

产出结构：
| ID | 模块 | 子模块 | 优先级 | 用例标题 | 前置条件 | 步骤 | 预期结果 | 用例类型 | 设计方法 | Tags | 关联需求 | 实际结果 | 执行人 | 结果 |
|----|------|-------|-------|---------|---------|-----|---------|---------|---------|------|---------|---------|-------|------|

**特性**：
- 首行表头冻结、加粗、填色
- 自适应列宽
- "优先级"列有下拉数据验证（P0/P1/P2/P3）
- "结果"列有下拉（Pass/Fail/Blocked/NA）
- "结果"列 Pass=绿色、Fail=红色条件格式
- 步骤和预期用换行区分多项（而非合并成一段）

### 备选方案：调用 `document-skills:xlsx`
当需求不是标准用例表（例如"给我一个测试进度跟踪表"、"Bug 统计表"等），直接调用 `document-skills:xlsx` skill，它能处理任意 Excel 需求。

### 导入禅道 / TestLink 格式
- **禅道**：用脚本加参数 `--format zentao`，会按禅道官方模板排列列顺序
- **TestLink**：生成 XML 而非 xlsx，通过 `--format testlink` 参数

## 5. XMind (.xmind) 输出

### 优先方案：调用本仓库脚本
```bash
python scripts/cases_to_xmind.py cases.yaml -o output/订单测试用例.xmind
```

生成的 .xmind 文件可**直接双击用 XMind 打开**（ZEN 2020+ 版本支持）。

### 思维导图结构
```
订单模块测试用例 v1.2.0 (根节点)
├─ 订单创建 (模块)
│  ├─ 正常下单 (子模块)
│  │  ├─ [P0] 使用默认地址+默认支付方式下单成功
│  │  │  ├─ 前置条件
│  │  │  │  └─ 已登录 + 购物车有商品
│  │  │  ├─ 步骤
│  │  │  │  ├─ 1. 进入购物车页
│  │  │  │  ├─ 2. 勾选商品并结算
│  │  │  │  └─ ...
│  │  │  └─ 预期
│  │  │     └─ ...
│  │  └─ [P1] 使用新地址下单
│  └─ 异常下单
│     └─ ...
└─ 订单支付
   └─ ...
```

### 优先级标记
节点上自动打颜色标签：
- P0 = 红色
- P1 = 橙色
- P2 = 黄色
- P3 = 蓝色

### 备选方案
如果 xmind-sdk 安装失败，退化到 "XMind 大纲"（见下节），用户手动导入。

## 6. XMind 大纲（Markdown 层级）

**零依赖方案**，输出 Markdown，用户粘贴进 XMind / MindMaster / 幕布即可成树。

```markdown
# 订单模块测试用例 v1.2.0
## 订单创建
### 正常下单
#### [P0] 使用默认地址+默认支付方式下单成功
##### 前置条件
- 已登录普通用户账号
- 购物车内有至少 1 件有库存商品
##### 步骤
1. 进入购物车页
2. 勾选 1 件商品，点击"去结算"
3. ...
##### 预期
- 跳转到支付页，显示订单号
- ...
### 异常下单
#### [P1] 库存不足时下单
...
```

**导入 XMind**：XMind → 文件 → 导入 → Markdown

## 7. Word (.docx) 输出

用于**正式文档**（测试方案、测试计划、测试报告、验收报告）。

### 优先方案：调用 `document-skills:docx`
测试方案/计划/报告是 `document-skills:docx` 的典型用例。触发时直接调用该 skill，给它下面这些素材：
- 标题、版本、作者、日期
- 章节结构（用 `references/test-design.md` §11 的测试计划模板为骨架）
- 具体内容
- 如果用户有公司模板（.docx），优先套用

### 产出要求
- 自动生成目录
- 一级标题使用"Heading 1"样式（便于后续生成目录）
- 表格使用 Word 原生表格（不要截图）
- 页眉/页脚带项目名和页码

## 8. PPT (.pptx) 测试报告

用于**团队汇报、迭代结束总结、验收评审**。

### 优先方案：调用 `document-skills:pptx`

### 推荐的标准幻灯结构

| 页 | 标题 | 内容要点 |
|---|------|---------|
| 1 | 封面 | 项目名 / 版本 / 汇报人 / 日期 |
| 2 | 测试范围 | 本次测试覆盖的模块/功能/不覆盖的 |
| 3 | 测试数据概览 | 用例总数、执行数、通过率、Bug 数（大数字可视化） |
| 4 | 用例执行情况 | 按模块的通过率柱状图 |
| 5 | Bug 分布 | 按严重度/模块的分布图 |
| 6 | 关键 Bug 列表 | Top 5 需要关注的 Bug |
| 7 | 遗留风险 | 未修复 Bug、已知限制、兼容性风险 |
| 8 | 性能测试结果（如有） | P50/P95/P99 + 吞吐量图 |
| 9 | 结论与建议 | 是否可上线、需关注事项 |
| 10 | Q&A | 感谢页 |

### 产出要求
- 封面简洁（避免文字堆砌）
- 数据页要有图表，不要只有表格
- 结论页用加粗、对比色突出核心观点
- 遵循公司已有 PPT 模板（如用户提供）

## 9. 多格式同时产出

用户在启动协议 Q3 中可多选。此时按以下顺序处理：

1. **先生成结构化中间数据**（JSON/YAML，不给用户，仅作为单一数据源）
2. **依次调用对应生成器**，产出多份文件
3. **告知用户所有产物的路径**，格式示例：

```
✅ 已为你产出以下文件：

📄 测试用例 Markdown：discuss/order-cases.md
📊 测试用例 Excel：docs/testing/订单测试用例.xlsx
🧠 测试用例 XMind：docs/testing/订单测试用例.xmind
🤖 自动化脚本：tests/api/test_order.py
🚀 执行脚本：scripts/test-order-api.js
```

### 依赖关系表

| 格式 | 依赖 | 回退方案 |
|-----|------|---------|
| Markdown | 无 | — |
| xlsx | `openpyxl` 或 `document-skills:xlsx` | Markdown 表格 |
| xmind | `xmind-sdk`（或手搓 zip） | XMind 大纲（Markdown） |
| docx | `document-skills:docx` 或 `python-docx` | Markdown |
| pptx | `document-skills:pptx` 或 `python-pptx` | Markdown 大纲 |

依赖不可用时：
1. 告知用户具体缺什么包
2. 给出安装命令（`pip install openpyxl` 等）
3. **同时提供回退方案的产物**（不让用户空手而归）
