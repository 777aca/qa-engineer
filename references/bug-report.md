## 1. Bug 定位思路

发现异常不等于发现 Bug。**定位**是从"现象"到"根因"的收敛过程：

1. **精确描述现象**：什么动作导致了什么结果？环境？账号？时间？
2. **最小化复现**：一步一步去掉无关操作，直到最短复现路径
3. **分层排查**：网络层？前端？后端？数据库？第三方？
4. **二分定位**：代码版本二分（git bisect）、数据二分（先一条再十条）
5. **假设验证**：每次只改一个变量
6. **给出根因**：不要停留在"点击按钮没反应"，要说"前端在 XHR 返回 401 时未正确跳转到登录页"

## 2. 分层排查清单

拿到异常后，沿栈自上而下排查：

### UI 层
- 元素是否渲染？`document.querySelector(...)` 存在吗？
- 控制台是否有 JS 报错？
- Network 面板里该请求发出了吗？状态码是什么？
- 状态管理（Redux/Vuex/Pinia）里的值对吗？

### 网络层
- 请求 URL、Method、Headers、Body 对吗？
- CORS 是否通过？preflight 是否成功？
- 是否有缓存干扰？尝试 `?_=${Date.now()}`
- 用 curl / Postman 能否复现？

### 服务端
- 请求是否到达了服务端？（看访问日志）
- 后端日志栈错误信息是什么？
- DB 查询是否返回了预期数据？手工执行 SQL 看看
- 依赖的第三方服务是否正常？

### 数据层
- 数据状态是否正确？是否有脏数据？
- 事务是否回滚？
- 索引缺失导致慢查询？

### 环境层
- 是否只在某浏览器/OS/设备上出现？
- 是否只在某地区/网络下出现？
- 时钟/时区是否正确？

## 3. 日志与证据收集

### 必收集项
- **截图或录屏**：标注问题区域
- **浏览器控制台**：Console 错误 + Network 面板（建议保存 HAR）
- **服务端日志**：按 trace_id / request_id 串联
- **环境信息**：被测版本、浏览器、OS、分辨率、账号
- **时间戳**：精确到秒，方便后端查日志

### HAR 导出
Chrome DevTools → Network → 右键 → Save all as HAR with content。

### 录屏工具
- 本机：Windows Xbox Game Bar / macOS 自带 / ffmpeg
- 浏览器：Playwright `video: 'on'`
- 移动端：scrcpy / 系统自带屏幕录制

## 4. 严重度与优先级

两个维度独立：

### 严重度（Severity）— 技术视角
| 等级 | 含义 |
|-----|------|
| S1 致命 | 系统崩溃、数据丢失、安全漏洞、核心功能不可用 |
| S2 严重 | 主要功能有缺陷但有 workaround |
| S3 一般 | 次要功能有缺陷，影响体验 |
| S4 轻微 | 文案、样式、提示不准确 |

### 优先级（Priority）— 业务视角
| 等级 | 含义 |
|-----|------|
| P0 立即 | 阻塞发布，立刻修 |
| P1 本版本 | 当前迭代内必须修 |
| P2 下版本 | 下一个迭代修 |
| P3 待定 | 择机修或不修 |

**注意**：高严重度不一定高优先级（例：某个很偏门的管理后台崩溃 = S1P2）。

## 5. Bug 报告标准模板

```markdown
## [Bug] {一句话结论，说清楚"在什么条件下，什么功能有什么异常"}

### 基础信息
- **报告人**：{姓名}
- **发现时间**：2026-04-21 14:32 GMT+8
- **严重度**：S2
- **优先级**：P1
- **所属模块**：下单 / 支付 / 登录 / ...
- **关联需求**：PRD-123 / JIRA-456

### 环境
- **被测版本**：v1.2.3 (git commit abc1234)
- **环境**：staging / https://staging.example.com
- **浏览器**：Chrome 138.0.7204.157 (Win11 23H2 1920x1080)
- **账号**：test+buyer01@example.com（普通用户）
- **网络**：4G / WiFi / 有线

### 复现步骤
1. 登录 test+buyer01 账号
2. 进入"我的订单"页
3. 点击订单号 O20260421001 右侧的"申请退款"按钮
4. 在弹窗中选择退款原因"商品质量问题"并提交

### 期望结果
- 弹窗关闭
- 订单状态变为"退款中"
- 列表刷新显示"退款中"标签
- 用户邮箱收到退款申请确认邮件

### 实际结果
- 弹窗关闭
- 页面白屏，Console 报错：`TypeError: Cannot read property 'status' of undefined at OrderList.tsx:142`
- 订单状态未更新（手动刷新后仍为"已完成"）
- 邮件未收到

### 必现/偶现
- 必现（5/5 次复现成功）

### 附件
- screenshot-01.png（报错截图）
- console.log（控制台输出）
- network.har（网络请求）
- backend.log（后端日志，trace_id=x-7f3a9c）

### 可能原因（可选，提供便于开发定位）
- 后端 `/api/refund/apply` 返回 200，但 body 为空字符串，前端 JSON.parse 失败
- 疑似缓存穿透或 feature flag 未配置

### 验证方式（fix 后请这么验证）
1. 以 buyer01 账号重复上述步骤
2. 另外抽测 2 个已完成订单的退款流程
3. 检查邮件到达
```

## 6. 常见反模式

❌ **标题过于模糊**：
- 差：`订单有 Bug`
- 好：`[Bug] 已完成订单申请退款后页面白屏，Console 报错 TypeError`

❌ **步骤省略关键细节**：
- 差：`登录后申请退款就崩了`
- 好：见上面模板，每一步可被复制粘贴执行

❌ **"期望"与"实际"混淆**：
- 分两段写，读者一眼看清差异

❌ **没有环境信息**：
- 只说"我这边不行"，结果是浏览器缓存/Chrome 扩展/本地代理问题

❌ **没附证据**：
- 纯文字描述 UI 问题，开发看不懂，来回拉扯浪费时间

❌ **夹带抱怨**：
- 不要写"这功能怎么能这样设计"，就事论事

## 7. 复现不稳定 Bug 的技巧

偶现 Bug 最难缠，但几乎都可以通过以下手段稳定复现：

### 策略 1：放大条件
- 网速：慢网（Chrome DevTools Network → Slow 3G）
- 数据量：造 1 万条数据
- 并发：脚本同时发 100 个请求
- 时间：setTimeout 延长 / 跨时区

### 策略 2：记录一切
- 全量开 DEBUG 日志
- 前端装 `why-did-you-render` 看渲染
- 后端开 SQL log

### 策略 3：尝试推测的诱因
- 多标签页切换导致 token 过期？
- 用户双击导致重复提交？
- localStorage 有旧数据？清空 → 复现 → 对比

### 策略 4：用脚本暴力复现
```python
for i in range(1000):
    run_test_case()
    if is_bug_appeared():
        save_environment(i)
        break
```

### 策略 5：对拍（diff testing）
- 新旧版本同操作对比输出
- 两个账号做同样操作对比

找到稳定复现路径后，再提 Bug——否则开发大概率打回"无法复现"。
